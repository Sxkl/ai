# AI-Native TDD — Test-First as Default (左移 LS-01)

## 核心原则

> **"测试是先于代码的合约；AI 写测试比写代码快。"**

在 AI-Native Pod 开发范式中，测试不再是"写完代码后的可选项"，而是**定义功能行为的合约**。AI Agent 应以测试准确性和效率远高于人类，因此 TDD 流程对 AI 而言边际成本极低，却能从根本上杜绝"代码正确但行为不符"的致命缺陷。

## 触发条件

当用户表述包含以下任一意图时，**必须**加载并遵循本规范：

- 实现/开发/写功能、需求、特性、接口、API
- 修复 bug、改逻辑、重构
- "写代码"、"加个接口"、"改一下这个功能"
- 任何涉及**新增或修改业务逻辑**的操作

不适用于以下场景（但建议自愿遵循）：
- 纯排版/样式微调（无逻辑变更）
- 配置文件修改（已通过 schema 约束）
- 文档修改

## 强制工作流：TDD 六步循环

AI Agent 在每次功能实现中**必须**严格按以下顺序执行，不得跳过或重排任何步骤：

| Step | 阶段 | 行为 | 退出条件 |
|------|------|------|----------|
| **1. 理解** | Understand | 分析需求，明确输入/输出/边界条件，确认测试框架和文件路径 | 产出测试清单（checklist of scenarios） |
| **2. 测试先行** | Red | **先写测试文件**（单元测试 + 必要时集成测试），覆盖正常路径 + 异常路径 + 边界值 + 空值 | 测试文件已创建，可编译/解析通过 |
| **3. 验证失败** | Verify Red | 运行测试，**确认测试失败**（证明测试能捕获缺失的实现） | 至少一个测试因"功能未实现"而失败 |
| **4. 最小实现** | Green | 编写**刚好让测试通过**的最小实现代码，不做过度设计 | 所有测试通过 |
| **5. 验证通过** | Verify Green | 运行全量测试套件，确认无回归 | `npm test` / `pytest` / `go test` 全部绿色 |
| **6. 重构** | Refactor | 在测试保护下重构实现，消除重复、改善可读性、优化性能 | 重构后测试仍全绿 |

> ⚠️ **Step 3 (Verify Red) 不可省略**：如果直接在 Step 2 后就写实现，等于"先猜实现再补测试"，本质上不是 TDD。跳过此步骤即违规。

## AI Agent 行为规则

以下规则为**硬约束**，违例将导致代码被拒绝：

### 文件生成顺序

```
正确顺序:  测试文件 → 实现文件
错误顺序:  实现文件 → 测试文件  ❌
```

1. Agent **必须先 generate/write 测试文件**，再 generate/write 实现文件。
2. 在单次对话中，若用户请求"实现功能 X"，Agent 的**第一个文件操作必须是测试文件**。
3. 若任务涉及多个模块，按模块粒度执行：Module A 测试 → Module A 实现 → Module B 测试 → Module B 实现。

### 对话输出约束

- Agent 在生成实现代码前，必须明确展示测试用例清单。
- Agent 在声称"完成"前，必须展示测试运行结果截图或输出。

## Eval Gate：定义"完成"

代码**不能**被认为"Done"直到满足以下全部条件：

| Gate | 条件 | 验证方式 |
|------|------|----------|
| G1 | 测试文件先于实现文件创建 | 检查文件时间戳 / git log |
| G2 | 所有测试通过 | 测试运行器输出 `0 failures` |
| G3 | 未跳过任何强制步骤 | 检查 Step 1-6 执行记录 |
| G4 | 覆盖率 ≥ 阈值（见下文 Harness） | `--coverage` 报告 |
| G5 | 无新增 lint 错误 | `eslint` / `ruff` / `golangci-lint` |
| G6 | 测试可在 CI 环境独立运行（不依赖本地状态） | 无 hardcoded path / env / 时间依赖 |

> 任一 Gate 未通过 → 回到对应 Step 修复，不得标记为完成。

## Test Harness 标准

### 测试框架选择

优先使用项目已有的测试框架。若项目无测试框架，Agent 需在 Step 1 (理解) 阶段与用户确认后初始化。

### 合格测试的构成要件

一个"合格的测试"必须同时满足：

| 维度 | 要求 |
|------|------|
| **类型安全** | 测试代码具有完整的类型标注（TypeScript: `strict` / Python: `mypy` / Go: 编译通过） |
| **覆盖率阈值** | 行覆盖率 ≥ 80%，分支覆盖率 ≥ 70% |
| **正常路径** | 至少 1 个 Happy Path 测试 |
| **异常路径** | 至少 1 个错误/异常输入测试 |
| **边界值** | 空值 `null`/`undefined`/`[]`/`""`、极值（`0`、`-1`、`MAX_INT`）、超长输入 |
| **隔离性** | 单测不依赖网络、数据库、文件系统（使用 mock/stub） |
| **可重复** | 同一测试多次运行结果一致（无随机性、无时间耦合） |
| **命名规范** | `{场景}_{输入}_{预期}` 如 `should_return_error_when_input_is_empty` |

### 集成测试额外要求

当需求涉及跨模块交互、数据库操作、或 API 调用时，除单元测试外还必须包含：

| 测试类型 | 应用场景 | 最低要求 |
|----------|----------|:--------:|
| API 契约测试 | REST/RPC 接口 | 1 个 success + 1 个 error 状态码 |
| 数据库测试 | 涉及 CRUD | 1 个 write + 1 个 read（可用 testcontainers / 内存DB） |
| 事件/消息测试 | Kafka/MQ/Event | 1 个 publish + 1 个 consume（用 mock broker） |

## 反模式（Anti-Patterns）：禁止的行为

以下行为**明确禁止**，发现即退回：

### AP1: 先写实现后补测试
```
❌ 写完所有业务代码 → 想起来要写测试 → 补几个测试应付
✅ 先写测试 → 确认失败 → 写实现 → 确认通过
```
**后果**: 测试成为"事后自证正确的工具"而非"事前约束行为的合约"，批量遗漏边界 case。

### AP2: 跳过边界/异常 case
```
❌ 只测 Happy Path："输入正常数据 → 返回 200"
✅ 同时测: null/undefined、超长字符串、SQL 注入字符、并发请求、超时
```
**后果**: 生产事故 90% 发生在边界条件。Happy Path 通过不代表代码正确。

### AP3: 过度 Mock 导致测试不测真实行为
```
❌ mock 一切 → 测试只验证 "mock 被调用了"
✅ mock 外部依赖 (DB、网络)，验证被测单元的真实逻辑输出
```
**判断标准**: 删掉实现代码后，测试是否仍然能通过？若能 → 测试无效。

### AP4: 为了覆盖率写无意义测试
```
❌ 测试 getter/setter / 纯数据结构 / 框架生成代码
✅ 测试业务逻辑、条件分支、状态转换、副作用
```
**判断标准**: 测试失败是否意味着"业务行为出错"？若否 → 这是凑数测试。

### AP5: 跳过 Verify Red（测试预期失败的步骤）
```
❌ 写完测试不运行，直接写实现 → 测试第一次运行就全绿
✅ 写完测试 → 运行 → 看到 RED → 写实现 → 看到 GREEN
```
**后果**: 无法确认测试是真的"测试了正确的东西"，还是"测了一个永远通过的空壳"。

### AP6: 一个大测试覆盖所有场景
```
❌ 一个 test() 里跑 10 个不相关的断言
✅ 每个独立场景一个 test()，失败时能快速定位
```
**判断标准**: 一个测试失败是否只需要看一个原因？若否 → 拆分。

## 语言/框架特定指引

### TypeScript / JavaScript

```
测试框架: Vitest / Jest
Mock: vi.mock() / jest.mock()
覆盖率: vitest --coverage
命名: *.test.ts / *.spec.ts
```

### Python

```
测试框架: pytest
Mock: unittest.mock / pytest-mock
覆盖率: pytest --cov --cov-report=term
命名: test_*.py
```

### Go

```
测试框架: testing (标准库)
Mock: testify/mock 或接口注入
覆盖率: go test -coverprofile=coverage.out
命名: *_test.go
```

### Java / Kotlin

```
测试框架: JUnit 5 + Mockito
覆盖率: JaCoCo
命名: *Test.java / *Test.kt
```

## 执行流程速查

```
用户: "帮我实现一个用户注册接口"
  │
  ├─ Step 1: 理解需求
  │   └─ 输出测试清单: [正常注册, 用户名已存在, 密码过短, 邮箱格式错误, 空字段]
  │
  ├─ Step 2: 先写测试文件 register.test.ts
  │   └─ 包含 5 个 test case
  │
  ├─ Step 3: 运行测试 → 5 failed (RED ✅ 确认测试有效)
  │
  ├─ Step 4: 写 register.ts 最小实现 → 运行测试 → 5 passed (GREEN ✅)
  │
  ├─ Step 5: 运行全量测试 → 无回归 (VERIFY ✅)
  │
  ├─ Step 6: 重构（提取验证逻辑/改善错误消息等）
  │   └─ 重构后测试仍全绿 (REFACTOR ✅)
  │
  └─ Eval Gate G1-G6 通过 → 标记 Done
```

## 例外与豁免

以下情况可在 Step 1 时声明豁免，但**必须在代码注释中标注原因**：

- **纯样板代码**（如 ORM Model 定义、proto 生成文件）：标记 `// tdd-skip: boilerplate`
- **紧急 Hotfix**（P0 生产事故）：可在修复后 24h 内补测试，需在 commit message 中注明 `[HOTFIX-TODO-TEST]`
- **探索性原型**（一次性验证、不会合并到主分支）：标记为 `// tdd-skip: prototype`

> ⚠️ 豁免不等于放松标准。Hotfix 必须在 follow-up ticket 中补测试，原型在正式分支上仍需 TDD。
