## Context
Redis distributed lock race condition: `get(key)` + `delete(key)` in non-atomic pattern. During RollingUpdate, two pods compete for lock, leading to `Unable to connect to Redis` on evicted pod.

## Solution Pattern
```lua
-- Lua script for atomic check-and-delete
if redis.call("GET", KEYS[1]) == ARGV[1] then
    return redis.call("DEL", KEYS[1])
else
    return 0
end
```

```java
// Java invocation
String script = "if redis.call('GET', KEYS[1]) == ARGV[1] then return redis.call('DEL', KEYS[1]) else return 0 end";
RedisScript<Long> redisScript = RedisScript.of(script, Long.class);
stringRedisTemplate.execute(redisScript, Collections.singletonList(lockKey), lockValue);
```

## Key Learnings
- **Never use get+delete pattern** for distributed locks — it's a classic anti-pattern
- Lua scripts execute atomically in Redis; no race window
- Verify lock ownership (value check) before deletion to avoid deleting another pod's lock
- Also relevant: SOP-002 (K8s RollingUpdate → Redis pool exhaustion)
