# 标准需求开发前期准备 Skill

## 适用场景

当用户提供**需求文档**并给出 **P4 编号**时，触发本 Skill，执行"开发前期准备"全流程：

1. 识别需求文档类型（自动判断，无需用户声明）。
2. 创建并关联任务（PR-XXXX）。
3. 基于远程 develop 创建 feature 分支。
4. 生成初版开发计划（plan）并等待审阅。
5. 审阅通过后提交 plan，并回填工时与评论记录。
6. 全流程严格串行执行，任一步失败立即停止，不执行后续步骤。

---

## 需求文档类型识别

在 Step 0 预检查阶段，**优先自动判断需求文档类型**，并在 Step 3 生成 plan 时应用对应模板。

### 类型定义

| 场景代码 | 触发特征 | 典型来源 |
|---------|---------|---------|
| `DOC_PRD` | 包含用户故事、业务流程、字段说明、验收标准等 | 产品 PRD、需求说明书 |
| `DOC_API` | 包含接口地址、请求参数、响应格式、鉴权方式、回调机制、沙箱/测试环境 | 第三方开放平台 API 文档、Swagger、Postman Collection |
| `DOC_PROTO` | 以 UI 截图、原型图、交互稿为主，文字描述为辅 | Figma、蓝湖、Axure 链接或截图 |
| `DOC_BUG` | 包含复现步骤、期望行为、实际行为、影响版本 | Bug 工单、测试报告 |
| `DOC_REFACTOR` | 包含架构图、模块边界、改造前后对比、兼容性说明 | 技术方案文档、架构评审结论 |
| `DOC_NONE` | 无正式文档，仅有口头描述、P4 标题或 Jira 评论 | 紧急需求、小改动 |

### 识别规则

1. 优先根据文档内容关键词自动判断，无需用户声明类型。
2. 若一份文档同时命中多个类型（如 PRD 中附带 API 接口说明），取**主导类型**，次要类型作为补充上下文。
3. 若无法自动判断，输出候选类型列表请用户确认，**确认前不进入 Step 1**。
4. 类型确认后，在所有后续步骤输出中标注当前场景代码（如 `[DOC_API]`）。

### 场景识别关键词速查

| 场景代码 | 高置信关键词 |
|---------|-----------|
| `DOC_PRD` | 用户故事、验收标准、业务流程、字段说明、产品需求 |
| `DOC_API` | API Key、endpoint、request/response、webhook、sandbox、签名、回调、鉴权 |
| `DOC_PROTO` | Figma、蓝湖、Axure、原型、UI 截图、页面跳转、交互稿 |
| `DOC_BUG` | 复现步骤、期望行为、实际行为、BUG、缺陷、影响版本 |
| `DOC_REFACTOR` | 重构、技术债、架构改造、模块拆分、性能优化、兼容性 |
| `DOC_NONE` | （无上述关键词，或输入仅为 P4 标题/一句话描述） |

## 输入约定

- 必填输入：
  - 需求文档（文本、链接、附件摘要、截图描述，支持所有文档类型）
  - P4 编号（示例：P4-5938）
- 默认规则：
  - 任务处理人 = 报告人 = 当前操作者
  - 任务完成时间 = 下周五
  - Start Date / Target Date 为空
  - 预计工时 / 剩余工时 初始为空，待 plan 审阅后回填

### Jira 字段映射（可配置模板）

优先使用以下映射，避免跨项目字段差异导致失败：

```yaml
jiraMapping:
  p4LinkField: "customfield_xxxx"
  estimateField: "timetracking.originalEstimate"
  remainingField: "timetracking.remainingEstimate"
  dueDateField: "duedate"
  assigneeField: "assignee"
  reporterField: "reporter"
```

字段缺失或不可写视为失败，立即停止流程。

### Jira 创建字段固定规则（PR 项目）

当项目为 `Engineering 4.x (PR)` 时，创建任务必须严格按以下字段：

- 项目：`PR`
- 问题类型：`任务`（强制映射 `issuetype.id=10113`）
- 概要：`<P4 概要> - 后端开发`
- 报告人：当前用户（默认）
- 截止时间：下周五（默认）
- 描述：创建时留空，plan 审阅通过后更新
- 优先级：`Medium`（默认）
- 链接的问题：`blocks` 指向输入的 P4（如 `P4-5220`）
- 经办人：当前用户（默认）
- 初始预估：留空，plan 审阅通过后更新
- 剩余的估算：留空，plan 审阅通过后更新
- Target start：留空
- Target end：留空
- 测试负责人：`yadong.liu@linksfield.net`（默认）

补充映射（来源于 Jira 创建页配置）：

- `PR`（projectId=`10264`）使用 issueType 配置组 `10695`
- 在配置组 `10695` 中：
  - `10113` = 任务
  - `10115` = 故事
  - `10116` = 故障
  - `10000` = Epic
- Jira QuickCreate（`/secure/QuickCreateIssue.jspa?decorator=none`）关键表单字段：
  - `pid=10264`
  - `issuetype=10113`
  - `priority=3`（对应 Medium）
  - `issuelinks=issuelinks`
  - `issuelinks-linktype=blocks`
  - `issuelinks-issues=P4-XXXX`
  - 仅当上述三项链接字段同时存在时，页面创建可一次性带出 linked issue。

## 执行步骤

### Step 0：预检查

1. **文档类型识别**：按"需求文档类型识别"章节自动判断场景代码，结果输出给用户确认（5 秒内无异议视为确认）。
2. 校验 P4 格式是否合法且可检索。
3. 校验 Jira 连接与认证是否可用（必须可创建任务）。
4. 校验 Jira 关键字段可写（P4 关联、处理人、报告人、Due Date）。
5. 检查远程 `origin/develop` 是否存在。
6. 若以上任一步失败，立即停止并输出失败原因，不执行 Step 1 及后续步骤。

### Step 1：创建任务并关联 P4

创建 Jira 任务并关联 P4，任务号必须由 Jira 创建结果返回（例如 `PR-6312`），并满足：

- 任务标题必须为"需求主题（P4-XXXX）- 后端开发"，不得出现"开发前期准备"等一次性阶段性措辞。
- 项目固定为 `Engineering 4.x (PR)`。
- 问题类型固定为 `任务`。
- 创建参数需显式使用 `issuetype.id=10113`（不可仅依赖文案"任务"）。
- 关联上游 P4，关联方式必须落在 Jira「链接的问题（Linked Issues）」中（使用 `Blocks` 关系）。
- 指定处理人、报告人（同一人）。
- Due Date 设为下周五。
- 优先级固定为 `Medium`。
- 预计工时、剩余工时留空。
- Start Date、Target Date 留空。
- 测试负责人默认 `yadong.liu@linksfield.net`。
- 创建时 description 保持空（不预填）。
- 若创建失败或未返回有效任务号，立即停止流程。
- 若创建后未检测到有效的「链接的问题」关联（PR-XXXX ↔ P4-XXXX），立即停止流程，不进入 Step 2。
- 若"问题类型=任务"等用户硬约束无法满足，必须立即停止并向用户反馈失败原因、接口报错原文、已尝试参数；禁止降级为"故事"等其他类型继续创建。
- 若使用 `issuetype.id=10113` 仍失败，必须停止并反馈，不得继续 Step 2。
- 若通过 API 尝试补链返回成功但 `linkedIssues(P4-XXXX)` 校验仍为 0，视为补链失败，必须停止并反馈。

输出内容：

- 任务号（Jira 实际返回）
- 任务链接
- 处理人 / 报告人
- Due Date
- 文档类型标注（如 `[DOC_API]`）

### Step 2：创建开发分支

仅使用 Step 1 返回的任务号创建分支，不允许使用 P4 编号推断任务号。

```bash
git fetch origin
git checkout -b feature/<Step1返回任务号> origin/develop
```

分支命名必须严格为：`feature/PR-XXXX`（其中 `PR-XXXX` 来自 Jira 创建结果）。
若目标分支已存在，视为流程冲突并立即停止（不复用历史分支）。

输出内容：

- 新分支名
- 分支来源（origin/develop）

### Step 3：生成初版 plan（待审阅）

基于需求文档 + 项目上下文 + **当前场景代码**，生成 `docs/plan` 下的计划文档：

- 路径：`docs/plan`
- 文件名：`YYYYMMDD_<Step1返回任务号>_plan.md`

#### 通用 plan 结构（所有场景必含）

- 目标与范围
- 影响模块与依赖
- 最小可执行任务拆解（任务粒度可独立交付/验证）
- Checklist（开发、联调、自测、回归）
- 工时估算（总工时 + 分项：开发 / 联调 / 自测 / 方案评审）
- 风险与回滚点

#### 场景差异化扩展结构

**`DOC_PRD`（标准 PRD）**

在通用结构基础上额外包含：

- 核心业务流程梳理（主流程 + 异常分支）
- 数据模型变更清单（新增 / 修改 / 废弃字段）
- 前后端接口约定（path / query / body / response 字段均使用 `snake_case`）
- 权限与角色影响范围
- 数据迁移方案（若涉及存量数据）

**`DOC_API`（第三方 API 集成）**

在通用结构基础上额外包含：

- 对接目标：第三方服务名称、接口版本、文档来源
- 鉴权方式：API Key / OAuth / HMAC 签名等，密钥存储方案（test/prod 双环境）
- 核心接口清单：

  | 接口名称 | 方法 | 路径 | 用途 | 是否异步 |
  |---------|------|------|------|---------|
  | 创建订单 | POST | /v1/payment | 发起支付 | 否 |
  | 支付回调 | POST | /webhook | 异步通知 | 是 |
  | 订单查询 | GET  | /v1/payment/{id} | 状态同步 | 否 |

- 回调处理方案：接收路径、验签逻辑、幂等设计、重试机制
- 沙箱验收计划：测试账号、测试金额、验收用例（正常支付 / 支付失败 / 重复回调）
- 环境配置清单（test / uat / prod 各环境 base_url、key 占位符）
- 异常处理：超时、重复通知、签名错误的降级策略
- 上线检查项：密钥切换、回调地址白名单、日志脱敏

**`DOC_PROTO`（设计稿 / 原型图）**

在通用结构基础上额外包含：

- 页面 / 组件清单（逐一列出，注明是新增还是改造）
- 交互行为说明（从截图/原型中提取的关键交互逻辑）
- 前后端接口需求列表（根据原型推断，需在联调前对齐）
- 响应式 / 多端适配要求
- 视觉验收标准（如需 UI 走查，注明节点）

**`DOC_BUG`（Bug 修复）**

在通用结构基础上额外包含：

- 问题根因分析（根因 + 触发路径）
- 修复方案（方案 A / B 对比，推荐方案及理由）
- 影响范围评估（是否影响存量数据、其他模块、已有接口）
- 回归测试范围（最小回归用例集）
- 热修复 vs 版本修复决策（若需区分）

**`DOC_REFACTOR`（重构 / 技术优化）**

在通用结构基础上额外包含：

- 改造前现状说明（现有问题、技术债描述）
- 改造后目标架构（模块边界、接口契约变化）
- 兼容性策略（向后兼容 / 强制升级 / 灰度切换）
- 拆分阶段与里程碑（按可独立发布的最小单元拆分）
- 性能基线 & 验收指标（如 P99 延迟、内存占用等）
- 回滚方案（feature flag / 双写 / 版本回退）

**`DOC_NONE`（无正式文档）**

使用通用结构，并在 plan 顶部强制插入以下声明：

```markdown
> **[文档缺失声明]**
> 本 plan 基于以下有限信息生成：`描述信息来源`
> 以下内容为推断，需在联调前与相关方对齐确认：
> - [ ] 需求范围确认（与产品/业务方）
> - [ ] 接口约定确认（与前端/调用方）
> - [ ] 验收标准确认（与测试负责人）
```

输出内容：

- plan 文件路径（可点击链接）
- 场景代码标注
- plan 预计工时（用于后续回填）
- 请用户审阅确认

### Step 4：审阅通过后提交与回填

用户确认后执行：

1. 提交 plan 到当前分支（提交仅包含 plan 文件）。
2. 更新 PR-XXXX：
   - 预计工时 = plan 预计工时
   - 剩余工时 = plan 预计工时
3. 将 plan 精简版写入 Jira description（覆盖原 description）：
   - 仅保留：目标与范围、任务拆解摘要、总工时与拆分、关键风险与回滚点
   - `DOC_API` 场景额外保留：核心接口清单、沙箱验收计划
   - `DOC_BUG` 场景额外保留：根因分析摘要、影响范围
   - 保持可读且可执行，避免粘贴完整长文
4. 追加备注（comment）：说明该 plan 的依据、范围、风险假设。
5. 将"计划编制耗时"写入 Jira 评论（固定模板）。

评论模板（固定）：

```text
开发计划已审阅通过：
- plan 文件：docs/plan/YYYYMMDD_PR-XXXX_plan.md
- 文档类型：场景代码，如 DOC_API
- 计划编制耗时：Xh
- 总预计工时：Yh（开发 Ah / 联调 Bh / 自测 Ch / 方案评审 Dh）
- 计划依据：文档类型对应依据，见下表
```

计划依据说明（按场景代码自动填入）：

| 场景代码 | 计划依据文案 |
|---------|-----------|
| `DOC_PRD` | 需求文档 + 评审结论 |
| `DOC_API` | 第三方 API 文档 + 沙箱环境验证 |
| `DOC_PROTO` | 设计稿 / 原型图 + 交互说明 |
| `DOC_BUG` | Bug 复现记录 + 根因分析 |
| `DOC_REFACTOR` | 技术方案文档 + 架构评审结论 |
| `DOC_NONE` | 口头描述 + 推断（待对齐确认） |

提交信息建议：

```text
PR-XXXX docs: add initial development plan [DOC_API]
```

### Step 4 执行门禁

仅在以下条件全部满足时执行提交与回填：

- 用户明确审阅通过。
- plan 中存在可解析"总预计工时"。
- 当前分支为 `feature/PR-XXXX`。
- `PR-XXXX` 与 Step 1 返回任务号完全一致。
- PR-XXXX 与 P4-XXXX 的「链接的问题」关联仍然存在。
- `DOC_NONE` 场景下，plan 顶部"文档缺失声明"的三项 checklist 已有至少一项被用户确认。

若任一条件不满足，仅输出差异与待处理项，不提交、不回填。

## 输出模板

每次执行本 Skill，按以下结构输出：

1. 基础信息
   - P4 编号
   - 文档类型（场景代码）
   - PR-XXXX（必须为 Jira 新建返回）
   - 分支名
2. 产物清单
   - plan 文件路径
   - plan 预计工时
3. 审阅状态
   - 待审阅 / 已通过
4. 后续动作
   - 是否已提交
   - 是否已回填工时
   - 是否已写入固定评论模板

## 质量约束

- 不跳过"先审阅再提交"。
- plan 必须可执行，禁止只有概念性描述。
- 分支与任务号强绑定：`feature/PR-XXXX`，且任务号必须来自 Jira 新建结果。
- PR 标题必须是"需求主题（P4-XXXX）- 后端开发"。
- PR 创建时 description 必须为空；审阅通过后再写入精简版 plan 描述。
- P4 关联必须通过「链接的问题」建立，缺失则流程失败。
- 当 Jira 字段校验失败时，必须 Fail-Fast：立即停止并反馈，不允许自行改用其他字段值继续推进。
- 工时回填数值必须来源于 plan，不手工臆测。
- worklog 默认不再自动写入；必须在 Jira 评论中记录计划耗时。
- 提交前必须通过变更白名单校验：仅允许本任务相关文件。
- Jira 更新后必须二次读取校验 `description` / `timetracking` / `issuelinks`。
- plan 提交前执行 `snake_case` 请求字段扫描，不通过则禁止提交。
- 工时必须包含总量与分项（开发/联调/自测/方案评审）并与 Jira 回填一致。
- 禁止用 P4 编号推断 PR 任务号（例如禁止 `P4-5220 -> PR-5220`）。
- 若无法创建 PR，流程必须立即停止，不允许继续建分支或生成正式 plan 文件。
- 交互文档中与前端约定的请求参数（path/query/body）与返回字段必须统一为 `snake_case`。
- 业务语义仍可使用 `groupCode` 描述，但接口字段命名需落为 `group_code`。
- `DOC_NONE` 场景下，plan 文件顶部必须包含"文档缺失声明"，不得省略。
- `DOC_API` 场景下，plan 必须包含沙箱验收用例（至少：正常支付、支付失败、重复回调三个用例），否则视为 plan 不完整，不允许进入 Step 4。
- `DOC_BUG` 场景下，plan 必须包含根因分析，禁止仅描述"修改 XX 代码"等无根因的修复方案。
