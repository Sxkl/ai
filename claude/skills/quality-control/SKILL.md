# 强制验证与质量控制

在报告"完成"或总结变更前，必须完成以下验证：

1. **运行 Linting**
   - JavaScript/TypeScript: `eslint . --quiet`
   - Python: `ruff check`
   - Go: `golangci-lint run`

2. **运行测试**
   - 如果存在测试，运行相关的单元测试或集成测试

3. **代码审查**
   - 对于重构或架构变更，执行额外的手动代码审查检查

> ⚠️ **重要**：只有通过所有验证后，才能报告"完成"

# 上下文与文件处理规则

## 大文件处理
- 对于超过 500 行的文件，始终使用 `offset + limit` 分块读取
- **禁止**依赖单次完整读取

## 重构前准备
- 在进行大规模重构前，首先在独立提交中清理：
  - 死代码
  - 未使用的导入/变量
  - 调试日志

## 任务拆分原则
- 每个任务保持在合理数量的文件内（建议 ≤8 个文件）
- 避免触发激进的上下文压缩

## 复杂任务处理
- 主动拆分为多个并行工作的子代理
- 每个子代理专注于独立模块
- 最后整合结果

## 上下文保持
- 始终为后续步骤保持足够的上下文

# 工具使用最佳实践

## 搜索结果验证
- 如果 `grep` 或搜索结果异常偏低，按目录或文件验证，并注意可能的截断

## 全面搜索
重命名函数、修改签名或修改 API 时，必须全面搜索：
- 直接调用
- 类型引用
- 字符串字面量
- 动态导入/requires
- 重新导出
- Barrel 文件
- 测试 mock

## 输出限制
- 始终考虑工具输出的字符/行数限制
- 必要时请求完整输出

## 命令执行后检查
- 执行 bash 命令后，检查：
  - 退出码
  - stdout/stderr
  - 验证实际效果

# 任务执行工作流

每个任务必须遵循以下步骤：

| 步骤    | 内容                                   |
| ------- | -------------------------------------- |
| 1. 理解 | 理解任务并规划步骤（包括风险评估）     |
| 2. 执行 | 以小而可验证的步骤执行变更             |
| 3. 总结 | 总结具体变更、验证结果和潜在风险       |
| 4. 说明 | 对于架构或重大变更，提供清晰的推理说明 |

# 通用指令

## 代码风格
- 与项目现有的代码风格、架构和约定保持一致
- 除非明确要求重构，否则不要改变现有风格

## 代码质量
- 优先选择清晰、显式的代码
- 避免过早抽象或"魔法"代码

## 透明度
- 始终透明：解释你做了什么、为什么做、以及下一步计划

## 上下文管理
- 如果上下文即将降级或关键信息可能丢失，提前警告并建议优化策略

# 执行标准

> **严格遵守以上所有规则**，即使默认倾向更简单的方案。
> 始终以最高的工程标准执行。
