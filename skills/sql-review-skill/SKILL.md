# SQL Review SKILL

# SQL 审查（安全 + 性能 + 可维护性）

## 目标

把一段或多段 SQL 做成“可上线/可回滚/可观测”的质量审查输出，覆盖：

- 正确性风险（结果集、去重、聚合、边界条件）
- 性能风险（索引、扫描、JOIN、排序、分页、统计信息）
- 安全风险（注入、越权、敏感字段暴露、误删误更）
- 并发与锁（事务隔离、锁范围、死锁概率）
- 可维护性（可读性、可测试性、可复盘）

你必须输出“结论 + 风险清单 + 改写SQL + 索引建议 + 验证步骤”。当信息不足时明确列出“待补充项”，但仍给出基于默认假设的建议方案。

## 输入要求（审查前你要先整理）

尽量从用户输入中提取并填充下列信息；缺失则标记为“待补充”：

最小必填：

- `dialect`
- `sql`

可选：

- `env`
- `purpose`
- `schema`
- `params`
- `constraints`
- `evidence`

```yaml
dialect: "MySQL|PostgreSQL|SQLServer|Oracle|SQLite|ClickHouse|Other"
env: "prod|staging|dev|unknown"
purpose: "read|write|ddl|migration|report|unknown"
sql: |
  -- 粘贴SQL（可多段）
schema:
  tables:
    - name: "table_name"
      primary_key: ["id"]
      indexes:
        - name: "idx_x"
          columns: ["colA","colB"]
      row_count_est: 0
params:
  style: "prepared|string_concat|unknown"
  examples:
    - name: "case1"
      values: { key: "value" }
constraints:
  max_runtime_ms: 0
  max_rows: 0
  allow_full_scan: false
evidence:
  explain: ""
  slowlog: ""
  sample_result: ""
```

## 审查规则（必须覆盖）

### 1) 正确性

- WHERE 条件是否遗漏（时间窗、租户/组织隔离、软删除标记）
- JOIN 是否引入行膨胀（1:N 误当 1:1；维表重复）
- 聚合是否正确（GROUP BY 字段、聚合口径、COUNT(\*) vs COUNT(col)）
- NULL 语义是否正确（NULL 比较、COALESCE、NOT IN 的 NULL 陷阱）
- DISTINCT 是否被滥用（用 DISTINCT 掩盖 JOIN 错误）
- 时间与时区口径是否一致（UTC/本地；日期截断）
- 任何创建新表，修改表结构的语句，表，列，索引任何一项是否缺少注释

### 2) 性能

- 是否能命中索引（谓词可 sargable；避免对索引列函数/隐式类型转换）
- 是否存在不必要的全表扫描/全索引扫描
- JOIN 顺序与驱动表是否合理（高选择性先过滤；避免大表驱动大表）
- 排序/分组是否导致大范围 filesort/hash aggregate
- 分页是否高 offset（建议 seek/基于游标）
- 子查询/IN/EXISTS 的可替换性（半连接 vs 相关子查询）
- 统计信息是否可能失真（需要 ANALYZE/VACUUM/更新 stats 的场景）

### 3) 写操作安全（UPDATE/DELETE/INSERT）

- 必须包含明确的 WHERE（避免误更新/误删除）
- 是否需要分批（按主键范围/limit batch）降低锁与日志压力
- 是否需要“先 SELECT 预览影响行数”
- 是否需要幂等性（重复执行不产生额外副作用）
- 是否需要回滚方案（备份/影子表/可逆变更）

### 4) 事务、锁与并发

- 隔离级别是否匹配业务一致性要求
- 是否存在长事务（锁持有过久、阻塞）
- 是否会触发锁升级/范围锁（间隙锁、next-key lock）
- 是否容易死锁（交叉更新顺序不一致）

### 5) 安全与合规

- 是否存在 SQL 注入风险（拼接 SQL、未参数化）
- 是否读取/输出敏感字段（手机号、身份证、Token、密钥等）
- 是否缺少租户隔离条件（tenant\_id/org\_id）
- 是否需最小权限原则（只读账号、禁止 DDL）

### 6) 可维护性

- 可读性（别名、缩进、逻辑分段）
- 可测试性（给出最小数据集与断言）
- 可观测性（建议增加 trace\_id/审计字段/执行耗时记录）

## 输出格式（你必须按此结构输出）

### 0) 审查结论

- 结论：{可上线 / 需修改后上线 / 禁止上线}
- 风险等级：{S0阻断 / S1高 / S2中 / S3低}
- 主要原因（≤3条）：{...}

### 1) 风险与问题清单

用表格列出所有问题，至少包含：

| 编号 | 类型 | 严重度 | 问题描述 | 影响  | 证据/定位              | 建议修复        |
| -: | :- | :-- | :--- | :-- | :----------------- | :---------- |
|  1 | 性能 | S1  | {…}  | {…} | {EXPLAIN/索引/估算行数…} | {改写/索引/限制…} |

严重度定义：

- S0：有数据安全/越权/误删误更/明显错误结果风险
- S1：大概率导致线上慢查询、锁冲突、资源打满
- S2：特定规模或边界条件下会出问题
- S3：风格或小优化

### 2) 建议改写 SQL（可直接替换）

提供“改写前/改写后”或只给“改写后（可直接运行）”。改写必须遵守：

- 不改变业务语义（除非明确指出原 SQL 语义有误，并给出修正理由）
- 若涉及写操作，必须给出“预览影响行数”的 SQL

```sql
-- 改写后 SQL
{...}
```

### 3) 索引与结构建议

按表列出建议，必须说明“为什么能加速/会影响什么”：

| 表        | 建议                             | 预期收益                | 代价/风险    |
| :------- | :----------------------------- | :------------------ | :------- |
| t\_order | 新增联合索引 (user\_id, created\_at) | where + order by 命中 | 写入变慢/占空间 |

如涉及大表变更，给出灰度/在线 DDL 建议（按数据库方言）。

### 4) 验证与上线步骤（必须给）

- 验证正确性：给出 2-3 个用例（输入条件 + 预期行数/关键字段断言）
- 验证性能：要求提供 EXPLAIN / EXPLAIN ANALYZE（若可用）对比
- 上线策略：是否需要分批、限流、低峰执行、超时与回滚
- 观测指标：慢查询阈值、锁等待、影响行数、错误率

### 5) 待补充信息（如有）

列出你为了更准确审查还需要的信息：

- 方言/版本
- 表结构与索引
- 估算行数与选择性
- 真实 EXPLAIN
- 参数示例与典型时间窗

## 快捷问法示例（用户怎么调用你）

1. “请审查这段 MySQL 查询是否能命中索引，并给出改写建议：<SQL>”
2. “这是一个 UPDATE，要上线到生产。请帮我做安全审查与回滚方案：<SQL>”
3. “这是慢查询的 EXPLAIN（附上），请定位瓶颈并给出索引方案：\<SQL + EXPLAIN>”
